# 权限控制页面 & 新建访问链接优化 — 需求/开发计划

> 版本: v3.0.0-draft  
> 日期: 2026-04-19  
> 状态: **待确认**

---

## 一、需求分析

### 1.1 现状

当前系统已有基础权限框架：

- **AccessLink** 模型：访问链接（token + 启停状态）
- **LinkPermission** 模型：每链接 × 每资源 → `can_view / can_create / can_update / can_delete`
- **后端 RESOURCES 清单**（9 项）：5 个页面级 + 4 个按钮级（仅任务相关）
- **前端路由守卫**：`page:*` 资源的 `can_view` 校验
- **AppHeader**：仅 3 处按钮使用了 `hasPermission` 控制显隐

**现存问题：**

1. 权限粒度不足 — 大量页面内按钮/操作未接入权限控制
2. 权限配置 UI 扁平 — 无模块/页面层级分组，9 个资源一个平铺表格
3. 新建链接流程缺乏引导 — 无法快速设置「只读」「编辑」等预设模板
4. 无批量勾选能力 — 不能按模块/页面整块切换权限

### 1.2 目标

1. **按页面/模块维度制定权限开关** — 资源清单从 9 项扩展为 ~40 项，覆盖全部可见/可点击元素
2. **每个按钮独立权限开关** — 每个页面的每个操作按钮注册为独立 resource
3. **批量勾选** — 支持「全选模块」「全选页面」「全选全部」三级快捷勾选
4. **新建链接优化** — 增加预设模板（只读/编辑/管理员）快速创建
5. **不影响现有功能** — 管理员 (`?admin=1`) 仍全权限；已有链接自动迁移新增资源

## 三、权限配置 UI 优化设计

### 3.1 新建访问链接弹窗优化

**现状**：仅输入名称。

**优化后**：

```
┌─ 新建访问链接 ────────────────────────────────┐
│ 链接名称: [___________________]                │
│                                                │
│ 权限预设:                                      │
│   ○ 只读访问（仅查看所有页面，无操作按钮）       │
│   ○ 编辑权限（可查看+编辑，不可删除）            │
│   ○ 完全权限（全部开启）                        │
│   ● 自定义（创建后手动配置）                     │
│                                                │
│               [取消]  [创建]                     │
└────────────────────────────────────────────────┘
```

- **只读**：所有 `page:*` 的 `can_view=true`，所有 `btn:*` 的 `can_view=true`，其余关闭
- **编辑**：所有 `can_view=true` + `can_create=true` + `can_update=true`
- **完全**：全部 `true`
- **自定义**：全部 `false`（同当前逻辑）

### 3.2 权限编辑面板重构

**现状**：扁平表格，9 行。

**优化后**：分模块手风琴 + 三级全选

```
┌─ 权限配置 ──────────────────────────────────────────────────────┐
│ [✓ 全选全部]                                                    │
│                                                                  │
│ ▼ 模块 A：任务收集               [✓ 全选模块]                    │
│   ▼ 📋 任务收集页                [✓ 全选页面]                    │
│     │ 资源               │ 查看 │ 新增 │ 修改 │ 删除 │          │
│     │ 📋 任务收集页       │  ✓  │  -   │  -   │  -   │          │
│     │ ➕ 新建收集         │  ✓  │  ✓   │  -   │  -   │          │
│     │ 👁 查看任务         │  ✓  │  -   │  -   │  -   │          │
│     │ ✏️ 编辑任务         │  ✓  │  -   │  ✓   │  -   │          │
│     │ ✦ 设为首选          │  ✓  │  -   │  ✓   │  -   │          │
│     │ ⏸ 启停收集          │  ✓  │  -   │  ✓   │  -   │          │
│     │ 🗑️ 删除任务        │  ✓  │  -   │  -   │  ✓   │          │
│   ▼ 📄 任务详情页                [✓ 全选页面]                    │
│     │ ...                                                        │
│                                                                  │
│ ▶ 模块 B：需求工时统计           [□ 全选模块]                    │
│ ▶ 模块 C：周期统计               [□ 全选模块]                    │
│ ...                                                              │
└──────────────────────────────────────────────────────────────────┘
```

**交互规则：**

1. **全选全部** → 所有资源所有开关 = true / false
2. **全选模块** → 该模块下所有资源所有开关 = true / false
3. **全选页面** → 该页面分组下所有资源所有开关 = true / false
4. **页面级 can_view=false** 时，该页面下所有按钮自动禁用且置灰
5. 按钮行的「不适用」开关（如查看按钮不需要「新增」开关）显示为 `—` 禁用态

### 3.3 哪些开关对每个资源有意义

为避免无意义的开关造成困惑，定义每个资源的**有效开关列**：

| 资源类型 | can_view | can_create | can_update | can_delete |
|----------|----------|------------|------------|------------|
| `page:*` | ✅ 页面可见 | — | — | — |
| `btn:*:create` | ✅ 按钮可见 | ✅ 可执行 | — | — |
| `btn:*:edit` | ✅ 按钮可见 | — | ✅ 可执行 | — |
| `btn:*:delete` | ✅ 按钮可见 | — | — | ✅ 可执行 |
| `btn:*:import` | ✅ 按钮可见 | ✅ 可执行 | — | — |
| `btn:*:export` | ✅ 按钮可见 | — | — | — |
| `btn:*:view` | ✅ 按钮可见 | — | — | — |

> 无意义的列在 UI 显示为 `—`，数据库存 false 即可。

## 五、后端改造

### 5.1 RESOURCES 清单扩展

`backend/src/routes/permissions.js` 中 `RESOURCES` 数组从 9 项扩展为 40 项，分组结构化。

### 5.2 新建链接时自动初始化

创建链接时，为所有 40 个资源创建 `LinkPermission` 记录，根据预设模板设置初始值。

### 5.3 已有链接迁移

提供迁移逻辑：已有链接若缺少新增的 resource 条目，在 `GET /api/permissions` 和 `GET /api/permissions/check/:token` 时自动补齐（lazy migration），默认 `false`。

### 5.4 批量更新 API 优化

`PUT /api/permissions/:id/permissions` 支持接收一次性全量 permissions 数组（当前已支持），前端批量勾选时一次发送全部变更。

### 5.5 资源分组元数据 API

`GET /api/permissions` 返回的 `resources` 增加分组信息：

```json
{
  "resources": [
    {
      "resource": "page:tasks",
      "label": "📋 任务收集页",
      "module": "tasks",
      "module_label": "任务收集",
      "group": "page",
      "valid_actions": ["view"]
    },
    {
      "resource": "btn:tasks:edit",
      "label": "✏️ 编辑任务",
      "module": "tasks",
      "module_label": "任务收集",
      "parent_page": "page:tasks",
      "group": "btn",
      "valid_actions": ["view", "update"]
    }
  ]
}
```

## 六、数据库变更

### 6.1 无需新建模型

现有 `LinkPermission` 模型完全满足：`link_id + resource + can_view/create/update/delete`。

只需在 `RESOURCES` 清单中增加新条目，创建链接时自动初始化。

### 6.2 迁移策略

- **不改表结构** — `resource` 字段 VARCHAR(100) 足够
- **旧资源更名** — `btn:create_task` → `btn:tasks:create`，`btn:edit_task` → `btn:tasks:edit`，`btn:delete_task` → `btn:tasks:delete`，`btn:stop_task` → `btn:tasks:toggle`
- **迁移脚本** — 在服务启动时或首次 GET 时自动执行一次 rename + 补齐

## 七、FillPage 权限特殊处理

FillPage 通过专属 token 访问（`/fill/:token`），不经过 AccessLink 权限体系。

**方案**：FillPage 内的按钮权限可选择以下两种之一：

- **方案 A（推荐）**：FillPage 不纳入 AccessLink 权限控制，管理员通过是否生成专属链接控制访问。页面内按钮对所有持链接用户可见。
- **方案 B**：为 FillPage 增加专属权限配置，绑定到 StaffFillLink 而非 AccessLink。实现复杂度高。

> **建议采用方案 A**，FillPage 按钮资源仍注册到 RESOURCES 中供管理员配置，但实际仅对通过 AccessLink 访问的用户生效。直接通过 `/fill/:token` 访问的用户不受限。

## 九、兼容性保证

| 场景 | 处理方式 |
|------|----------|
| 管理员模式 (`?admin=1`) | `isAdmin=true` 时 `hasPermission` 始终返回 `true`，不受影响 |
| 已有链接旧资源名 | 启动时自动 rename（如 `btn:create_task` → `btn:tasks:create`） |
| 已有链接缺少新资源 | 查询时 lazy migration 补齐，默认 `false` |
| FillPage 专属链接 | 不受 AccessLink 权限限制 |
| 前端 `hasPermission` 查不到资源 | 返回 `false`（已有逻辑），按钮默认隐藏 → 安全 |
