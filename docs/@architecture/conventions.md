# DevTracker — 项目规约 & AI 开发铁律

> **版本**：v1.0 | **日期**：2026-04-13  
> **文档编号**：D02 | **性质**：**强制约束文档** — 所有开发活动（含 AI 辅助开发）必须严格遵守

---

## 一、文档体系规约

### 铁律 1：文档即真理 (Single Source of Truth)

| 编号 | 规则 | 违反后果 |
|:---|:---|:---|
| **1.1** | `requirements.md` 是唯一的需求基准，所有开发工作必须可追溯到某条 REQ 编号 | 无 REQ 对应的代码视为无效 |
| **1.2** | 本文档 `conventions.md` 是唯一的开发规约基准 | 不符合规约的变更必须回退 |

### 铁律 2：变更必须归档

| 编号 | 规则 |
|:---|:---|
| **2.1** | 对话中任何涉及**增加、删除、修改**功能/需求的内容，必须在**当轮对话结束前**同步更新到 `requirements.md` |
| **2.2** | 文档变更必须在变更记录表中登记：日期 + 变更内容 + 关联 REQ 编号 |
| **2.3** | 新增需求必须分配新的 REQ 编号，不得复用已有编号 |
| **2.4** | 删除的需求不物理删除，标记为 ~~删除~~ 并注明原因 |

### 铁律 3：三轮对话强制回溯

| 编号 | 规则 |
|:---|:---|
| **3.1** | 每经过 **3 轮对话**，AI 必须执行一次**文档回溯检查**：①重读 `requirements.md` ②重读 `conventions.md` ③检查当前工作是否与文档一致 |
| **3.2** | 回溯结果需记录：`[回溯✅] 第N次回溯，文档一致` 或 `[回溯⚠️] 发现偏离：...` |
| **3.3** | 发现偏离必须**立即修正**，先修改代码/文档使其一致，再继续开发 |

## 二、开发流程规约

### 铁律 4：模块完成制 — 做一个完一个

```text
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│  编码     │───→│  验证测试 │───→│  浏览器验证│───→│  标记完成 │
│  Module X │    │  Module X │    │  确认通过  │    │  [x]     │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↑                               │ 失败
      └───────────────────────────────┘
```

| 编号 | 规则 |
|:---|:---|
| **4.1** | **禁止跳模块开发**。当前模块的验证未通过前，不得开始下一个模块 |
| **4.2** | Demo 阶段以**浏览器验证**替代单元测试；正式版必须补充自动化测试 |
| **4.3** | 每个功能变更必须在浏览器中截图/录屏验证 |
| **4.4** | 验证必须包含：正常交互 + 边界情况 |

### 铁律 5：先文档后开发 — 开发准入门控

| 编号 | 规则 |
|:---|:---|
| **5.1** | **禁止未批准开发**：正式编码之前，必须先输出完整文档（需求、规约、设计方案） |
| **5.2** | **用户批准门控**：文档和 Demo 方案必须经用户明确批准后，才可进入正式编码阶段 |
| **5.3** | **Demo 优先**：先构建 Demo 原型验证可行性，Demo 验收通过后再进入正式开发 |
| **5.4** | **阶段性评审**：每个里程碑完成后，暂停并请用户评审，获得批准后才进入下一个里程碑 |

## 三、编码规约

### 铁律 6：代码质量底线

| 编号 | 规则 | 示例 |
|:---|:---|:---|
| **6.1** | **禁止魔法值**，使用命名常量 | ❌ `if x > 7` → ✅ `if x > DAYS_PER_WEEK` |
| **6.2** | **单例优先**，服务类组件使用单例模式 | `const Store = { ... };` |
| **6.3** | **DDD 分层**，代码按领域划分而非技术层划分 | `store.js`(数据), `matcher.js`(匹配), `components.js`(视图) |
| **6.4** | **接口隔离**，模块间通过清晰的函数签名交互 | `Matcher.matchRecords(records, staff)` |
| **6.5** | **错误处理**，关键操作必须有空值检查 | `if (!container) return;` |
| **6.6** | **操作反馈**，使用 Toast 通知用户操作结果 | `Utils.showToast('创建成功', 'success')` |

### 铁律 7：架构约束

| 编号 | 规则 |
|:---|:---|
| **7.1** | Demo 阶段技术栈锁定：**原生 HTML/CSS/JS**，不引入框架 |
| **7.2** | 正式版计划迁移至 **Vue 3 + Node.js**，Demo 架构需预留扩展性 |
| **7.3** | 数据层（Store）与视图层（Components）严格分离，通过 App 层协调 |
| **7.4** | CSS 采用 CSS Variables 设计系统，确保全局一致性 |

## 四、AI 辅助开发专项约束

### 铁律 8：防幻觉机制

| 编号 | 规则 |
|:---|:---|
| **8.1** | **禁止编造 API**：使用任何浏览器或第三方 API 前，必须确认存在且签名正确 |
| **8.2** | **禁止假设功能存在**：不得假设某个 DOM API 或 CSS 属性在所有浏览器可用 |
| **8.3** | **禁止跳过浏览器验证**：每个功能模块完成后必须在浏览器中实际验证 |
| **8.4** | **禁止占位符代码标记完成**：`// TODO`、空函数不计入完成度 |

### 铁律 9：行为边界

| 编号 | 规则 |
|:---|:---|
| **9.1** | **不得自行删除功能**：未经用户确认不得删除或降级需求文档中的功能 |
| **9.2** | **不得擅自更换技术栈** |
| **9.3** | **不得忽略用户反馈**：用户提出的修改意见必须纳入需求或规约文档 |
| **9.4** | **不得虚报测试结果** |
| **9.5** | **不得跳过错误处理**：遇到代码错误必须修复后再继续 |

### 铁律 10：对话纪律

| 编号 | 规则 |
|:---|:---|
| **10.1** | 每轮对话开始时，确认当前在 task.md 中的位置 |
| **10.2** | 每 3 轮对话执行铁律 3 的回溯检查 |
| **10.3** | 对话中提到的需求变更，在**本轮对话内**完成文档更新 |
| **10.4** | 遇到不确定的技术决策时，列出选项和利弊，由用户决定 |

## 五、个性化规约

### 铁律 11：版本号管理

| 编号 | 规则 |
|:---|:---|
| **11.1** | 版本格式：**语义化版本号** `vMajor.Minor.Patch`（如 v0.2.0） |
| **11.2** | Demo 阶段为 **v0.x.x**，正式版从 **v1.0.0** 开始 |
| **11.3** | Major：架构重大变更；Minor：新增功能模块；Patch：Bug 修复 |
| **11.4** | 每次发布必须在变更记录中登记版本号、日期和变更摘要 |

### 铁律 12：统计汇总规范

| 编号 | 规则 |
|:---|:---|
| **12.1** | 同一需求多个同岗位人员，在汇总表中**同一单元格水平并列显示** |
| **12.2** | 个人统计按**人员按钮平铺**（按角色分组），不使用下拉框 |
| **12.3** | 个人数据展示采用**可折叠的周期区块**，默认收起 |
| **12.4** | 任务列表按**年度→季度→月份**三级层级分组展示 |

### 铁律 13：新建任务交互规范

| 编号 | 规则 |
|:---|:---|
| **13.1** | 新建收集任务弹窗采用**左右双栏布局**：左侧表单 + 右侧日历 |
| **13.2** | 日历与表单**双向实时联动** |
| **13.3** | 如果日历选择的范围超出当前维度范畴，时间维度显示为"其他" |
| **13.4** | 日历下方显示**附近周期快捷选择列表** |
| **13.5** | 日历默认高亮今天，支持月份切换导航 |
