## 1. 本次开发铁律

1. 只修改本次需求涉及范围：设置页自动任务提示历史、钉钉 webhook 通知接收人配置、团队人员手机号维护、对应后端接口与数据字段。
2. 不修改任务收集、需求工时统计、周期统计、填报、权限控制等未提及业务行为。
3. 钉钉通知仍只使用群机器人 webhook，不恢复钉钉私聊、扫码登录、桌面客户端自动化或 Playwright 钉钉网页自动化。
4. 本次 @ 人功能只在“钉钉 webhook 通知开关开启”且“接收人弹窗 @ 功能开关开启”时生效；否则保持原固定文本发送。
5. 新增字段必须兼容旧数据：旧规则未配置接收人时继续可发送原文本；旧人员未配置手机号时显示为空。

## 3. REQ-35 自动任务提示历史删除修复与提示文案调整

### 3.1 问题

设置页自动任务历史提示展开列表中，部分记录点击右侧 `×` 后不能被删除，尤其包括：

- 校验类提示。
- 旧的自动执行日志类提示，例如“无需新增”记录。

### 3.2 要求

- 单条历史记录右侧 `×` 必须可删除该条记录。
- 批量删除必须同时支持校验消息和自动执行日志。
- 全部删除必须清空当前规则下全部提示历史和执行日志。
- 删除成功后页面立即刷新，不继续显示已删除记录。
- 删除失败时提示“删除失败”。

### 3.3 文案调整

将所有自动任务无法新增时的提示：

`本周任务已存在，若需新增，请手动处理`

统一改为：

`该任务已存在或无法新增超过下一周的新收集任务，若仍需新增，请手动处理`

覆盖范围：

- 后端自动任务服务返回文案。
- 自动执行日志文案。
- 校验提示历史文案。
- 设置页历史展示中来自旧文案的数据，前端展示时兼容替换为新文案。

## 4. REQ-36 团队人员手机号维护

### 4.1 用户故事

作为管理员，我需要在团队人员页面维护成员手机号，后续用于钉钉群机器人 webhook @ 对应人员。

### 4.2 页面要求

在团队人员页面的“姓名”列后新增一列：

- 列名：手机号码。
- 值为空时显示为空。
- 新增/编辑人员弹窗中新增“手机号码”输入项。
- 手机号码可为空，便于后续手动补充。
- 若填写手机号，保存时做基础格式校验：只允许数字，长度 5-20 位。

### 4.3 后端要求

`staff` 表新增字段：

- `phone VARCHAR(30) NULL`

接口调整：

- `GET /api/staff` 返回 `phone`。
- `POST /api/staff` 支持保存 `phone`。
- `PUT /api/staff/:id` 支持修改 `phone`。

## 5. REQ-37 钉钉 webhook 接收人配置与 @ 人发送

### 5.1 入口

在设置页每条规则的 webhook 通知区域中，在“测试发送”和“保存通知”按钮之间新增按钮：

`接收人`

点击后打开大弹窗。

### 5.2 弹窗布局

弹窗要求：

- 左上角标题：`webhook配置通知对象`。
- 右上角开关：控制 @ 人功能是否生效，默认关闭。
- @ 人功能开关打开后，在开关后方显示 @ 策略单选项：`仅@人`、`@所有人`，二选一。
- @ 策略默认值为 `仅@人`，兼容已保存的旧接收人配置。
- 开关关闭时，保存后仅发送原固定文本，不 @ 人。
- 开关打开且选择 `仅@人` 时，保存后测试发送和实际自动执行通知都按已勾选接收人 @ 人。
- 开关打开且选择 `@所有人` 时，保存后测试发送和实际自动执行通知都 @ 群内所有人，不再依赖勾选人员手机号。
- 右下角按钮：`保存编辑`、`退出`。

### 5.3 团队人员列表

弹窗内展示团队人员列表，字段包括：

- 多选框。
- 姓名。
- 角色。
- 手机号码。

默认勾选规则：

- 有手机号码的团队人员默认勾选。
- 手机号码为空的团队人员默认不勾选。
- 手机号码为空的团队人员不可勾选。
- 用户尝试勾选手机号码为空的行时提示：`手机号码为空，请在【团队人员】页面添加后再试`。

### 5.4 额外接收人

弹窗下方允许添加不在团队人员名单中的其他接收人。

每个额外接收人字段：

- 姓名。
- 角色。
- 手机号码。
- 勾选状态。
- 删除操作。

规则：

- 点击添加后新增一行可编辑接收人。
- 姓名、角色、手机号码均填写完整且手机号格式有效后，默认勾选。
- 额外接收人信息不写入团队人员表，只保存在当前自动任务规则的接收人配置中。

### 5.5 数据保存

`auto_task_rules` 表新增字段：

- `dingtalk_recipients TEXT NULL`

保存结构建议：

```json
{
  "enabled": true,
  "at_mode": "people",
  "staff_ids": ["staff-id-1"],
  "extra": [
    {
      "id": "local-id",
      "name": "张三",
      "role": "产品",
      "phone": "13800138000",
      "selected": true
    }
  ]
}
```

### 5.6 发送规则

发送钉钉 webhook 时：

- 如果规则的 webhook 通知开关关闭，不发送通知。
- 如果接收人 @ 功能开关关闭，保持原文本发送。
- 如果接收人 @ 功能开关打开且选择 `仅@人`，则收集已勾选且有手机号的团队人员与额外接收人手机号。
- 如果接收人 @ 功能开关打开且选择 `@所有人`，则请求体使用 `at.isAtAll = true`。
- 请求体增加钉钉 `at.atMobiles` 字段。
- 文本内容前追加 `@手机号`，保证钉钉群内实际触发 @。

请求体示例：

```json
{
  "msgtype": "text",
  "text": {
    "content": "@13800138000 自动任务已生成"
  },
  "at": {
    "atMobiles": ["13800138000"],
    "isAtAll": false
  }
}
```

@所有人请求体示例：

```json
{
  "msgtype": "text",
  "text": {
    "content": "@所有人 自动任务已生成"
  },
  "at": {
    "isAtAll": true
  }
}
```

### 5.7 验收标准

- 团队人员页面姓名后出现手机号码列。
- 新增/编辑人员可以维护手机号。
- 设置页 webhook 通知区域出现“接收人”按钮。
- 接收人弹窗默认关闭 @ 人开关。
- @ 人功能开关打开后，可以在 `仅@人` 和 `@所有人` 两种模式之间切换。
- 开关打开后，有手机号人员默认勾选，无手机号人员不可勾选。
- 可添加额外接收人，信息完整后可勾选。
- 保存后再打开弹窗，接收人配置保持。
- 测试发送时，若 @ 功能开启，请求体包含 `atMobiles`。
- 自动执行通知时，若 @ 功能开启，请求体包含 `atMobiles`。
- 测试发送和自动执行通知时，若选择 `@所有人`，请求体包含 `isAtAll: true`。
- @ 功能关闭时，原固定文本发送不受影响。
- 历史提示中单条、批量、全部删除均可删除截图中红框类型记录。

## 6. REQ-38 钉钉 webhook 卡片式通知

### 6.1 最终口径

用户已确认：不再实现“点击更新进度后识别点击者并跳转到自己的专属链接”的能力。

本需求仅实现：

- 钉钉群通知采用卡片式展示。
- 保留截图风格中的标题、正文和通知对象展示，不再显示底部“更新进度”按钮。
- 不做点击者身份识别。
- 不做按人跳转。
- 不新增钉钉企业应用、免登、OAuth、互动卡片回调等能力。

### 6.2 发送方式

基于当前群自定义机器人 webhook，发送体从普通 `text` 文本升级为钉钉 `markdown` 卡片式通知：

- `msgtype`: `markdown`
- `title`: `语音产研进度维护通知：`
- `text`: 使用 Markdown 组织通知正文和 @ 信息；通知内容输入框中的回车在发送前自动转换为 Markdown 硬换行，确保钉钉内按输入换行展示。

说明：

- 本版本不配置卡片底部按钮。
- 不允许将团队人员专属链接写入通知内容或卡片跳转参数。
- 不允许通过通知参数携带 staffId、fillToken 或可推测个人身份的信息。

### 6.3 内容组成

卡片内容来自当前设置页的“通知内容”输入框。

发送时自动组合：

- 固定标题：`语音产研进度维护通知：`
- 通知内容正文：管理员在设置页保存的固定文本。
- 正文换行：管理员在通知内容中输入的回车应在钉钉消息内可见，不被折叠为同一段空格。
- 通知对象：
  - 仅 @ 人：显示并发送 `@手机号`。
  - @ 所有人：显示并发送 `@所有人`。
- 不显示底部按钮。

### 6.4 与 @ 功能关系

卡片式通知必须继续兼容 REQ-37：

- @ 功能关闭时，只发送卡片正文。
- @ 功能开启且选择 `仅@人` 时，发送 `at.atMobiles`。
- @ 功能开启且选择 `@所有人` 时，发送 `at.isAtAll = true`。

### 6.5 验收标准

- 测试发送时，webhook 请求体为 `msgtype: markdown`。
- 自动执行通知时，webhook 请求体为 `msgtype: markdown`。
- 卡片标题为 `语音产研进度维护通知：`。
- 卡片不显示底部“更新进度”按钮。
- 通知内容中单次回车在钉钉内应显示为换行。
- 卡片不包含任何人员专属 fillToken。
- 仅 @ 人模式继续带 `atMobiles`。
- @ 所有人模式继续带 `isAtAll: true`。

## 7. 开发方案

### 7.1 后端

1. `Staff` 模型新增 `phone` 字段。
2. `staff` 路由新增/编辑时读写 `phone`，并做可空格式校验。
3. `AutoTaskRule` 模型新增 `dingtalk_recipients` 字段。
4. 自动任务 payload 归一化时保存接收人配置。
5. `sendDingTalkWebhook` 根据接收人配置生成 `atMobiles` 并发送。
6. 提示历史删除统一走按规则删除接口，确保 message 与 run log 都可删。
7. 迁移脚本补充 `staff.phone` 与 `auto_task_rules.dingtalk_recipients`。
8. `sendDingTalkWebhook` 使用 Markdown 卡片式通知，并保留 `at` 字段。

### 7.2 前端

1. `PersonnelPage.vue` 新增手机号列和表单输入。
2. `SettingsPage.vue` 增加接收人按钮与弹窗。
3. 弹窗加载 `/api/staff` 团队人员列表。
4. 保存时将接收人配置写入当前规则。
5. 测试发送和规则保存 payload 带上接收人配置。
6. 历史提示删除改为按规则传 ids 删除，并删除后立即刷新。

### 7.3 验证

必须验证：

- 后端语法校验。
- 前端生产构建。
- 本地页面浏览器验证：团队人员手机号列、接收人弹窗、历史删除。
- 接口验证：保存手机号、保存接收人配置、删除 message/run log。
